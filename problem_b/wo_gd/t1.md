# Problem B1: Security and Data Exposure Review

## 1. Internal ID Leakage
- **Attack/Failure Path:** Internal user IDs are exposed in API responses (e.g., `/users`, `/users/lookup`, backend helpers). Attackers or frontend clients may enumerate or misuse these IDs.
- **Impacted Endpoints/Helpers:**
  - `/users` (returns `id` for each user)
  - `/users/lookup` (returns `user_id`)
  - Backend helpers: `get_user_list`, `lookup_by_email`
- **Mitigation:**
  - Restrict exposure of internal IDs to trusted/internal clients only.
  - For frontend endpoints, return only non-sensitive identifiers (e.g., email, name).
  - Document and enforce separation between internal and external API consumers.

## 2. Authentication Boundaries
- **Attack/Failure Path:** Endpoints require bearer tokens, but test coverage shows possible gaps (e.g., missing token returns 401, but no test for token validation/expiry).
- **Impacted Endpoints/Helpers:**
  - All `/users/*` endpoints (display, header, list, lookup)
- **Mitigation:**
  - Ensure all endpoints consistently require and validate JWTs.
  - Add tests for token expiry, invalid tokens, and privilege escalation attempts.
  - Consider role-based access control for sensitive endpoints (e.g., admin-only actions).

## 3. Sensitive Data Exposure
- **Attack/Failure Path:** API responses may leak sensitive fields (e.g., `id`, `role`) or omit proper filtering. Some endpoints intentionally exclude fields (e.g., `/users/{id}/display` omits `id`/`role`), but others do not.
- **Impacted Endpoints/Helpers:**
  - `/users` (returns `role` and `id`)
  - `/users/{id}/display` (correctly omits sensitive fields)
- **Mitigation:**
  - Audit all response schemas for sensitive field leakage.
  - Use dedicated response models for public vs. internal endpoints.
  - Document intended exposure for each endpoint.

## 4. Input Handling and Validation
- **Attack/Failure Path:** Email lookup and user ID endpoints rely on direct input; malformed or missing parameters return 422/404, but no explicit validation for email format or user ID type.
- **Impacted Endpoints/Helpers:**
  - `/users/lookup` (email param)
  - `/users/{id}/display`, `/users/{id}/header` (user_id param)
- **Mitigation:**
  - Add stricter validation for email format and user ID type.
  - Return consistent error messages for invalid input.
  - Log and monitor failed lookups for abuse detection.

## 5. URL/External Call Safety
- **Attack/Failure Path:** Backend helpers make external requests to `api.example.com` without timeouts or error handling (see skipped test for timeout/malformed response). This could lead to denial of service or information leakage if the external API is compromised.
- **Impacted Endpoints/Helpers:**
  - `get_user_display`, `get_user_list`, `format_user_for_header`, `lookup_by_email` (backend helpers)
- **Mitigation:**
  - Implement timeouts and error handling for all external requests.
  - Validate and sanitize all data received from external APIs.
  - Limit external API usage to trusted sources and document fallback behavior.

---

**Summary:**
- Main risks: internal ID leakage, inconsistent sensitive data exposure, incomplete input validation, and lack of robust external call safety.
- Mitigations: restrict sensitive fields, enforce authentication, validate inputs, and harden external API interactions.

# Task 2: Security and Data-Exposure Concerns

## 1. Data Exposure Risks
- **User IDs**: The helpers `get_user_list` and `lookup_by_email` return user IDs. The PR constraint is not to expose internal user IDs to the front end. If these helpers are used server-side only, this is acceptable, but if any are exposed to the UI, it would violate the constraint.
- **User Data**: The code exposes `name`, `email`, and `role` fields. No sensitive fields are exposed, but if the API adds new fields in the future, they could be leaked unless explicitly filtered.
- **Email Exposure**: User emails are returned by several helpers. If these are shown in the UI, it could be a privacy concern depending on the application's audience and requirements.

## 2. Input Validation
- **User ID and Email**: Inputs to `get_user_display`, `format_user_for_header`, and `lookup_by_email` are not validated. The PR states that authentication and input validation are handled upstream, so this is in line with the scope. However, if this assumption changes, lack of validation could lead to issues.
- **API Response Validation**: The code does not validate the structure or content of API responses before accessing keys. This could lead to exceptions or data leakage if the API returns unexpected data.

## 3. Injection and URL Risks
- **URL Construction**: User IDs and roles are directly interpolated into URLs for API requests. If these values are not sanitized (even though validation is assumed upstream), there is a risk of injection or malformed requests if the upstream validation fails.
- **Email Lookup**: The email parameter is not used in a URL, so there is no direct injection risk here.

## 4. In/Out of Scope Considerations
- **Authentication**: Out of scope for this PR; assumed handled upstream.
- **Session Handling**: Out of scope.
- **Refactoring and API Contract**: Out of scope, so only the new helpers are relevant.

## Recommendations
- Ensure that helpers returning user IDs are not exposed to the UI.
- Explicitly filter and whitelist fields returned to the UI to prevent accidental data leakage if the API changes.
- Add comments or assertions to clarify the intended usage context (server-side vs UI).
- Consider sanitizing or validating user input even if upstream validation is assumed, to provide defense in depth.
- Validate API responses before accessing keys to avoid leaking unexpected data or causing exceptions.

---

**Summary:**
The main risks are accidental exposure of internal user IDs and user emails, lack of response validation, and potential URL injection if upstream validation fails. These should be addressed to align with the PR's constraints and best security practices.

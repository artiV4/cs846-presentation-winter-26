# Task 4: High-Risk Areas and Review Focus

## High-Risk or Complex Areas
1. **API Error Handling and Timeouts**
   - All user data is fetched via external API calls, but the code lacks error handling and explicit timeouts. This is high-risk for dashboard reliability and responsiveness.
2. **Data Exposure and Privacy**
   - Helpers return user IDs and emails. If these are exposed to the UI, it could violate privacy or the PR's constraint against exposing internal IDs.
3. **Performance and Redundant API Calls**
   - `format_user_for_header` makes two API calls per header render, which could cause latency or rate-limiting issues under load.
4. **Assumptions About Upstream Validation**
   - The code assumes all inputs are validated and authenticated upstream. If this assumption fails, there could be injection or data-leakage risks.

---

## Review Should Focus On
- **API Call Robustness**: Ensure all API calls have proper error handling and a 2-second timeout, as required by the PR constraints.
- **Data Filtering**: Confirm that only whitelisted fields (`name`, `email`, `role`) are ever exposed to the UI, and that internal IDs are never leaked.
- **Performance in Hot Paths**: Look for unnecessary or redundant API calls, especially in functions used frequently by the dashboard (e.g., header rendering).
- **Response Validation**: Check that API responses are validated before accessing keys to prevent exceptions or data leaks.

---

## Review Should Avoid/Deprioritize
- **Authentication and Session Handling**: Out of scope per the PR; these are handled upstream.
- **Refactoring Unrelated Modules**: Only review the new helpers, not unrelated code.
- **Changing External API Contracts**: The review should not suggest changes to the external user API itself.
- **Formatting or Style-Only Changes**: Unless they affect correctness, style changes are out of scope.

---

**Summary:**
Focus review on error handling, data exposure, and performance in the new helpers. Avoid unrelated refactoring, authentication/session logic, and external API contract changes, as these are out of scope for this PR.

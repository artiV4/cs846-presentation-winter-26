# Task 1: Code Review for Problem B PR

## Summary
This review covers the changes in `problem_b/user_helpers.py` as described in `test/PR.md`. Feedback is provided on correctness, error handling, security (UI exposure), and performance, with comments tied to the PR's constraints.

---

### 1. Correctness
- **Functionality**: The helpers implement the described features: user display lookup, user list retrieval, header formatting, and email lookup. The logic matches the PR summary.
- **Potential Issues**:
  - `get_user_display` and `get_user_list` assume the API always returns valid JSON and required keys (`name`, `email`, `id`). If the API response is malformed or missing keys, this will raise exceptions.
  - `format_user_for_header` fetches all users to find the role for a single user, which is inefficient and may not scale.

### 2. Error Handling
- **API Failures**: No error handling for network errors, timeouts, or non-200 responses. This violates the PR constraint: "API calls must timeout after 2 seconds" and risks unhandled exceptions.
- **Missing Data**: No checks for missing keys in API responses. This could cause `KeyError` or `TypeError` if the API changes or returns incomplete data.

### 3. Security (UI Exposure)
- **Exposure of Internal IDs**: `get_user_list` and `lookup_by_email` return user IDs. The PR constraint is not to expose internal user IDs to the front end. If these helpers are used server-side only, this is acceptable, but if any are exposed to the UI, it would violate the constraint.
- **Data Filtering**: Only `name`, `email`, and `role` are exposed in user lists, which aligns with the constraint. However, the code does not explicitly prevent leaking other fields if the API adds them in the future.

### 4. Performance
- **Unnecessary API Calls**: `format_user_for_header` calls both `get_user_display` and `get_user_list`, resulting in two API calls for a single header render. This could be optimized by combining data or caching.
- **Latency Risks**: No timeout is set on `requests.get`, risking dashboard blocking on slow API responses. This violates the 2-second timeout constraint.

---

## Recommendations
- Add error handling for network failures, timeouts, and bad responses (e.g., use `try/except`, check `r.status_code`, set `timeout=2`).
- Validate API response structure before accessing keys.
- Avoid exposing internal user IDs to the UI; clarify usage context in docstrings.
- Optimize `format_user_for_header` to avoid redundant API calls.
- Consider caching or batching user data if used in hot paths.

---

**Constraint Mapping:**
- **No internal IDs to UI**: Ensure helpers are not directly exposed to the front end.
- **2s API timeout**: Add `timeout=2` to all `requests.get` calls.
- **Dashboard responsiveness**: Reduce redundant API calls and handle slow responses gracefully.
- **Input validation**: Assumed handled upstream, but still validate API responses.

---

Overall, the helpers implement the required features but need improved error handling, explicit timeouts, and performance optimizations to fully meet the PR's constraints.

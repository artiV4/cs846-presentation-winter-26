# Test and Edge-Case Suggestions for problem_b/user_helpers.py (PR: test/PR.md)

**Step 1 — Context summary**

- **Intent of the PR:**
  - Integrate the User API to display the logged-in user's name and role in the dashboard header.
  - Add helpers for user display and lookup, including support for resolving users by email (for internal/support tool use).
- **Affected components:**
  - `problem_b/user_helpers.py` (helpers: `get_user_display`, `get_user_list`, `format_user_for_header`, `lookup_by_email`).
- **High-risk/complex areas:**
  - API integration (external calls, error handling, timeouts, and data exposure to UI).
  - Ensuring internal user IDs are not exposed to the frontend.
  - Performance (avoiding unnecessary API calls in the dashboard hot path).

**Step 2 — Test and Edge-Case Suggestions**

Reviewing for: correctness, error handling, security, and performance as highlighted in the PR.

---

### Edge Cases and Error Conditions to Test

1. **Network/API failures:**
   - The external User API is unreachable (network down, DNS failure).
   - The User API returns a non-200 HTTP status (e.g., 500, 404, 403).
   - The User API call times out (should enforce 2s timeout as per PR constraint).

2. **Malformed or incomplete API responses:**
   - The API returns an empty response or an empty list.
   - The API returns JSON missing required keys (e.g., `name`, `email`, `id`, `role`).
   - The API returns invalid JSON (malformed, not parseable).

3. **User not found scenarios:**
   - `get_user_display` is called with a user_id that does not exist.
   - `lookup_by_email` is called with an email that does not match any user (should return `None`).
   - `format_user_for_header` is called with a user_id not present in the user list.

4. **Role filtering and edge cases:**
   - `get_user_list(role=...)` is called with a role that matches no users (should return an empty list).
   - Users with missing or unexpected role values (e.g., role is `None`, missing, or an unknown string).

5. **Performance and repeated calls:**
   - Multiple rapid calls to `format_user_for_header` (test for redundant API calls and latency).

6. **Data exposure and privacy:**
   - Ensure that internal user IDs are not exposed to the frontend in any output.
   - Ensure that email addresses are only shown where appropriate (per user role or privacy policy).

7. **Input edge cases (internal use):**
   - `lookup_by_email` is called with an invalid email format (should not crash).
   - `get_user_display` is called with a non-integer or unexpected user_id type.

---

**Assumptions:**
- Authentication and input validation are handled upstream (API gateway).
- This module is not responsible for unrelated modules or architectural changes.

**Non-goals:**
- No architectural redesign, new frameworks, or unrelated refactoring.
- No style-only changes unless they affect correctness or error handling.

# Security and Data-Exposure Review for problem_b/user_helpers.py (PR: test/PR.md)

**Step 1 — Context summary**

- **Intent of the PR:**
  - Integrate the User API to display the logged-in user's name and role in the dashboard header.
  - Add helpers for user display and lookup, including support for resolving users by email (for internal/support tool use).
- **Affected components:**
  - `problem_b/user_helpers.py` (helpers: `get_user_display`, `get_user_list`, `format_user_for_header`, `lookup_by_email`).
- **High-risk/complex areas:**
  - Data returned from external API, exposure of user information to the UI, and handling of user identifiers.

**Step 2 — Security and Data-Exposure Review**

Reviewing for: data exposure, input validation, injection/URL risks, and alignment with PR constraints.

---

### 1. Data Exposure Risks
- **Internal user IDs:**
  - The PR requires that internal user IDs (UUIDs, PKs) are not exposed to the frontend. `format_user_for_header` returns only name and role, which is correct for the dashboard header.
  - However, `get_user_list` returns a list of user dicts including the `id` field. If this output is ever passed to the frontend/UI, it could expose internal user IDs, violating the PR's constraint. Ensure only display-safe fields are sent to the UI.
- **Email addresses:**
  - Both `get_user_display` and `get_user_list` return user email addresses. If these are shown in the UI, consider whether this is appropriate for all users (e.g., privacy concerns for non-admins).
- **Role information:**
  - Role is included in the output. This is likely intended, but ensure that exposing role information does not leak sensitive privilege details.

### 2. Input Validation
- **User IDs:**
  - The PR states that authentication and input validation for user IDs are handled upstream (API gateway). The module assumes valid, authenticated requests, so lack of input validation here is acceptable per scope.
- **Email in `lookup_by_email`:**
  - No explicit validation of the email parameter. If this is only used internally (as stated), risk is low, but if ever exposed to user input, consider validating email format to prevent misuse.

### 3. Injection and URL Risks
- **URL construction:**
  - `get_user_display` and `get_user_list` build URLs by concatenating user input (user_id, role) directly into the URL string. If user_id or role are not sanitized, this could allow for crafted input to manipulate the request URL (e.g., path traversal, injection). However, since input validation is handled upstream, and user_id/role are assumed safe, this is acceptable per PR scope. Still, using parameterized requests or explicit type checks is a best practice.
- **No use of query parameters dict:**
  - The code appends `?role=...` directly to the URL string. Using the `params` argument in `requests.get` would be safer and more robust against injection or encoding issues.

### 4. Other Security Considerations
- **No authentication or session logic in this module:**
  - Per PR, authentication is handled upstream. No secrets or tokens are handled here.
- **No direct database access:**
  - This module does not manage database transactions or direct DB queries.

---

**Summary of recommendations:**
- Ensure that outputs from `get_user_list` are not passed directly to the frontend if they contain internal user IDs or sensitive fields.
- Consider whether exposing user email addresses in the UI is appropriate for all user roles.
- For future-proofing, prefer using `requests.get(..., params={...})` for query parameters to avoid injection risks.
- If the usage of `lookup_by_email` ever expands beyond internal tools, add email format validation.

**Assumptions:**
- All input validation and authentication are handled upstream (API gateway).
- This module is not responsible for unrelated modules or architectural changes.

**Non-goals:**
- No architectural redesign, new frameworks, or unrelated refactoring.
- No style-only changes unless they affect correctness or security.

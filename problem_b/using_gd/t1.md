### Security and Trust-Boundary Risks

**1. Internal ID Leakage**
- **Attack/failure path:** Internal user IDs are exposed in API responses and backend helpers, which could allow enumeration or privilege escalation.
- **Impacted function(s):**
    - `get_user_list` (lines 61–75, `user_helpers.py`): returns `id` for each user.
    - `lookup_by_email` (lines 89–101, `user_helpers.py`): returns internal `id`.
    - Endpoint `/users/lookup` (lines 172–181, `user_helpers.py`): exposes internal `id`.
- **Mitigation:**
    - Restrict exposure of internal IDs to trusted/internal clients only.
    - Use opaque/public IDs for frontend exposure.
    - Document and enforce separation between internal and external API consumers.

**2. Sensitive Data Exposure**
- **Attack/failure path:** User roles and internal IDs are returned in API responses, potentially revealing privilege structure or sensitive information.
- **Impacted function(s):**
    - `get_user_list` (lines 61–75, `user_helpers.py`): returns `role` and `id`.
    - Endpoint `/users/{user_id}/header` (lines 142–171, `user_helpers.py`): exposes role in display string.
- **Mitigation:**
    - Ensure only authenticated users can access these endpoints.
    - Consider role-based access control for listing roles or restricting role visibility.

**3. Authentication Boundaries**
- **Attack/failure path:** Endpoints may be accessed without valid authentication, leaking user data.
- **Impacted function(s):**
    - All new endpoints (lines 102–181, `user_helpers.py`): use `Depends(auth.get_current_user)` for authentication.
- **Mitigation:**
    - Confirm that authentication is enforced by FastAPI dependency injection.
    - Maintain tests for unauthenticated requests (401 responses).

**4. URL/External-Call Safety**
- **Attack/failure path:** Backend helpers make requests to external API URLs using user-supplied data, risking SSRF or data leakage if not properly sanitized.
- **Impacted function(s):**
    - `get_user_display` (lines 39–44, `user_helpers.py`): uses `requests.get`.
    - `get_user_list` (lines 46–52, `user_helpers.py`): uses `requests.get`.
    - `format_user_for_header` (lines 54–60, `user_helpers.py`): uses `requests.get`.
    - `lookup_by_email` (lines 89–101, `user_helpers.py`): uses `requests.get`.
- **Mitigation:**
    - API base URL is fixed and trusted; user input is only used in path/query parameters.
    - Validate/sanitize user input before constructing URLs if base URL ever becomes user-configurable.
    - Implement timeouts and error handling for all external requests.
